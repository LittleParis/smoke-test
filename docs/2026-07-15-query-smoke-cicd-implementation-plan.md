# 查询接口冒烟自动化与 CI/CD 门禁实施计划

## 执行状态（2026-07-15）

- 阶段 0—8 的框架、配置、测试、文档和 CI/CD 门禁改造已完成。
- 已增加本地 HTTP Mock 环境，覆盖 CSRF 登录、Cookie 会话和三条 YAML 查询流程。
- Ruff、全量单元测试、YAML 门禁收集和 Swagger 扫描收集均已通过。
- 真实测试环境端到端执行待本地 `.env` 或 CI Secrets 配置完成后验收；仓库不再保存默认账号密码。

## 1. 目标

在现有项目基础上建设一套稳定、可扩展的查询接口冒烟测试框架，并作为 CI/CD 发布门禁运行。

第一阶段完成后，应达到以下效果：

- CI/CD 部署测试环境后，自动执行核心查询业务流程。
- 任意核心步骤失败时，pytest 返回非零退出码，流水线立即中断。
- 新增业务流程时，原则上只需要增加或修改 YAML，不需要编写新的执行代码。
- 测试报告能准确展示失败流程、失败步骤、请求参数、响应内容和断言原因。
- Swagger 批量扫描继续保留，但不作为发布门禁，只用于手动或定时补充覆盖。

## 2. 本期范围

### 2.1 本期实现

- 查询接口的多步骤业务编排。
- 前序响应字段向后续接口传递。
- YAML 加载、格式校验和引用校验。
- HTTP 状态及通用 JSON 字段断言。
- 查询接口安全保护。
- Complaint、Repair、Accident 三个模块的核心查询流程。
- 单元测试、真实环境验证和 CI/CD 门禁接入。
- Allure 结果及现有 HTML 报告输出。

### 2.2 本期不实现

- 新增、修改、删除等写操作流程。
- 测试数据创建、回滚和自动清理。
- 条件分支、循环、并行步骤等复杂编排语法。
- 完整的失败通知改造；通知作为后续独立阶段处理。
- 用冒烟测试替代完整接口回归测试。

## 3. 目标架构

```text
业务 YAML
   ↓
配置加载与启动前校验
   ↓
查询安全检查
   ↓
流程执行器
   ├── HTTP 客户端
   ├── 流程上下文与变量引用
   ├── JSON 路径提取
   └── 通用断言引擎
   ↓
步骤结果与 Allure 附件
   ↓
pytest 退出码
   ↓
CI/CD 继续或中断
```

设计原则：

1. 框架负责请求、传值、断言、安全和报告。
2. YAML 只负责描述业务接口的执行顺序和期望结果。
3. 核心业务流程的数据来源必须显式可见，主要通过“列表查询 → 提取字段 → 详情查询”获得。
4. YAML 核心流程不再依赖隐藏的模块预取缓存；原有预取机制仅保留给 Swagger 批量扫描模式。
5. 所有引用缺失、字段提取失败和配置错误都必须明确失败，禁止静默删除参数后继续请求。

## 阶段 1：确定并校验 YAML 契约

### 工作内容

- 为流程配置增加明确的 schema 版本，例如 `version: 1`。
- 明确流程级字段：`name`、`module`、`enabled`、`tags`、`steps`。
- 明确步骤级字段：`name`、`endpoint`、`method`、`params`、`extract`、`expect`。
- 启动前校验以下错误：
  - 流程名、模块名或步骤列表缺失。
  - 步骤名称重复。
  - endpoint 不是项目内 API 路径。
  - HTTP 方法不在允许范围内。
  - `$ref` 引用了不存在的步骤或字段。
  - extract、expect 的类型错误。
- 对现有 `prefetch` 和 `$cache` 提供清晰迁移策略：旧配置可以暂时加载，新编写的核心流程不再使用。

### 主要文件

- `core/pipeline_loader.py`
- `config/pipelines/README.md`
- `tests/unit/test_pipeline_loader.py`

### 完成标准

- 错误 YAML 在发送网络请求前直接失败，并指出文件名和具体字段。
- 合法 YAML 能稳定转换成内部配置对象。
- 原有合法配置不会无提示失效。

## 阶段 2：重构流程上下文和参数引用

### 工作内容

- 为每条 pipeline 创建独立的运行上下文，禁止流程间共享业务数据。
- 支持两类引用：
  - `$var.platform`：引用框架注入的公共运行变量。
  - `$ref.<step>.<field>`：引用前序步骤提取的数据。
- 核心流程取消对隐藏 `prefetch_cache` 的依赖。
- 引用没有命中时立即判定当前步骤失败，不再静默删除该参数。
- 保持参数解析逻辑与请求发送逻辑分离，便于单元测试。
- 暂时只要求引用覆盖查询参数中的完整值，不扩展字符串模板拼接等复杂语法。

### 主要文件

- `core/pipeline_loader.py`
- `core/pipeline_executor.py`
- `tests/test_pipeline.py`
- `tests/unit/test_pipeline_executor.py`

### 完成标准

- 列表接口提取的 ID 能稳定传入详情接口。
- 缺失引用会给出“哪个步骤、哪个参数、哪个引用未命中”。
- 两条流程连续或并行收集时不会串用上下文数据。

## 阶段 3：统一 JSON 路径与断言能力

### 工作内容

- 抽离统一的 JSON 路径读取能力，支持：
  - `result.id`
  - `result.items.0.id`
  - `result.items[0].id`
- extract 中声明的字段默认视为必需字段；路径不存在时步骤失败。
- 支持第一阶段必需的通用断言：
  - `status`：HTTP 状态码。
  - `equals`：字段等于固定值或引用值。
  - `exists`：字段必须存在。
  - `not_empty`：字符串、对象或列表不能为空。
- 对售后系统的标准响应提供默认断言：`isSuccess=true`、`errorCode=0`，同时允许 YAML 明确覆盖。
- 断言错误必须包含 JSON 路径、期望值和实际值。

### 主要文件

- 新增 `core/json_path.py`
- 新增 `core/assertion_engine.py`
- 修改 `core/pipeline_executor.py`
- 新增对应单元测试

### 完成标准

- 列表为空、详情 ID 不一致、业务状态失败时均能正确阻断流程。
- 报错信息不需要查看框架源码即可定位问题。

## 阶段 4：增加查询安全保护

### 工作内容

- YAML 核心门禁默认只允许 GET。
- 保留 GET 副作用接口黑名单，命中黑名单时禁止执行。
- 如果以后确实存在安全的 POST 查询接口，必须加入显式只读白名单后才能运行。
- 增加运行环境保护，只允许在明确配置的测试/演示环境执行。
- BASE_URL、登录地址、账号密码通过环境变量或 CI Secrets 注入；源码不再提供真实默认密码。
- 报告和日志对 Password、Cookie、Authorization、Token 等字段进行脱敏。

### 主要文件

- 新增 `core/safety_guard.py`
- 修改 `config/settings.py`
- 修改 `.env.example`
- 修改 `core/pipeline_executor.py`
- 修改报告附件生成逻辑

### 完成标准

- 写操作或黑名单接口无法被误加入发布门禁。
- 未声明为测试环境时拒绝执行真实冒烟请求。
- 日志、Allure 结果和 HTML 报告中没有明文密码或认证信息。

## 阶段 5：完善步骤结果和报告

### 工作内容

- 每个步骤记录：步骤名、接口、耗时、解析后的参数、状态、提取字段和失败原因。
- 任一步骤失败后，当前流程立即停止，并保留此前成功步骤的结果。
- 区分以下失败类型：
  - 配置失败。
  - 安全检查失败。
  - 登录失败。
  - 网络或超时失败。
  - HTTP 状态失败。
  - 业务状态失败。
  - 数据提取失败。
  - 断言失败。
- Allure 中以流程为测试用例、步骤为子步骤展示。
- 现有 `generate_report.py` 能继续读取结果并显示核心失败信息。

### 主要文件

- `core/pipeline_executor.py`
- `tests/test_pipeline.py`
- `generate_report.py`
- `tests/unit/test_pipeline_executor.py`

### 完成标准

- 测试失败时，可以直接从报告判断失败发生在哪一步以及为什么失败。
- 报告生成失败不改变冒烟测试本身的退出状态。

## 阶段 6：落地三条真实查询业务流程

### Complaint

```text
查询投诉列表 → 提取投诉 ID → 查询投诉详情 → 校验 ID 和业务状态
```

### Repair

```text
查询维修列表 → 提取 repairId → 查询维修详情 → 校验 repairId 和业务状态
```

### Accident

```text
查询事故列表 → 提取 accidentId → 查询事故详情 → 校验 accidentId 和业务状态
```

### 工作内容

- 使用真实 Swagger 参数名和响应路径编写 YAML。
- 每条流程只选择最能代表业务存活状态的核心接口。
- 列表接口必须断言存在可用数据，避免后续详情步骤使用空值。
- 不把导出、大数据量统计或不稳定接口放入第一批发布门禁。
- 为每条 YAML 增加配置加载测试，防止语法回归。

### 主要文件

- `config/pipelines/complaint.yaml`
- 新增 `config/pipelines/repair.yaml`
- 新增 `config/pipelines/accident.yaml`
- 对应配置测试

### 完成标准

- 三条流程在测试环境可以独立运行。
- 任意一条失败时，整体 pytest 执行返回非零退出码。
- 新增第四条同类流程时不需要修改执行器代码。

## 阶段 8：接入 CI/CD 发布门禁

### 建议流水线顺序

```text
安装依赖
→ ruff 代码检查
→ 框架单元测试
→ 部署测试环境
→ 查询业务冒烟
→ 成功后进入后续发布阶段
```

### 工作内容

- CI 使用明确的测试环境地址和 Secrets。
- 执行 `pytest tests/test_pipeline.py` 时保持默认失败退出行为，不配置 `continue-on-error`。
- 后续发布 job 依赖冒烟 job 成功，冒烟失败则不再执行。
- 测试结果和 HTML 报告使用 `always()` 保存，确保失败时也能查看。
- 手动执行入口继续保留，方便排查环境问题。
- Swagger 全量扫描放在手动触发或定时任务中，不阻塞常规发布。

### 主要文件

- `.github/workflows/smoke-test.yml`
- 如果领导实际使用其他 CI 平台，则提供等价命令和退出码约定

### 完成标准

- 人为制造一个断言失败后，流水线能正确中断后续阶段。
- 测试通过时，后续阶段正常执行。
- 无论成功或失败，都能下载测试报告。

## 阶段 9：稳定性验证与交付

### 验证矩阵

| 场景 | 预期结果 |
|---|---|
| 合法 YAML、接口正常 | 流程通过 |
| YAML 字段错误 | 请求前失败并指出字段 |
| 引用不存在 | 当前步骤失败，不发送错误请求 |
| 列表为空 | `not_empty` 断言失败 |
| 详情 ID 不一致 | `equals` 断言失败 |
| 接口返回 HTTP 500 | 流程失败，记录响应 |
| 接口返回 `isSuccess=false` | 流程失败，记录业务信息 |
| 配置写操作接口 | 安全检查拒绝执行 |
| 冒烟失败 | CI/CD 后续发布阶段中断 |
| 报告生成 | 成功和失败场景均有报告 |

### 交付物

- 可复用的 YAML 流程执行器。
- YAML 配置规范和示例。
- Complaint、Repair、Accident 三条核心流程。
- 完整单元测试。
- CI/CD 门禁配置。
- 本地运行、故障定位和新增流程说明。

### 最终验收标准

- 核心代码检查和单元测试全部通过。
- 三条查询业务流程在目标测试环境通过。
- 新增同类业务流程只需要编写 YAML。
- 任意核心流程失败都能阻止后续发布。
- 报告能够定位到具体流程、步骤和断言。

## 5. 后续阶段：失败通知

通知不阻塞本期地基建设，待发布门禁稳定后独立完善：

- 复用现有 `scripts/notify_wecom.py`。
- 只在失败或配置为始终通知时发送。
- 通知包含模块、流程、失败步骤、错误摘要和报告链接。
- 通知发送失败不能覆盖原始冒烟测试结果。
- Webhook 只从 CI Secrets 获取。

## 6. 实施纪律

- 严格按阶段顺序实施，每个阶段验证通过后再进入下一阶段。
- 每次只修改当前阶段涉及的文件，不做无关重构。
- 框架能力优先通过单元测试验证，真实环境只验证最终业务链路。
- 每次改动保留现有 Swagger 扫描能力，避免一次性替换造成回归。
- 发现接口响应结构与 Swagger 不一致时，先记录证据，再调整 YAML 或框架规则。
